\begin{abstract}
We considered switching the PPG12 cross-section normalization from the
current 60~cm-vertex-cut luminosity ($\Lint^{60\text{cm}}$, paired with a
\texttt{vertex\_eff} factor whose denominator is truth photons within
$|z_{\text{vtx}}^{\text{truth}}|<60$~cm) to the no-vertex-cut luminosity
($\Lint^{\text{noVtx}}$, paired with a vertex efficiency that uses ALL
truth photons in the denominator). The two conventions should be
mathematically equivalent in the single-interaction limit. In practice,
they differ by a factor of $\sim 3$ at 1.5~mrad and $\sim 2$ at 0~mrad,
because of a quirk in the MC vertex reweighting that affects the
no-vertex-cut denominator but not the 60-cm-cut denominator. We document
the issue, the per-$\pT$ comparison, and recommend keeping the current
convention until the MC vertex reweighting is reviewed.
\end{abstract}

\section{The Two Conventions}
\label{sec:conventions}

\textbf{Original (current production):}
\begin{equation}
  \frac{d\sigma}{d\pT} = \frac{N_{\text{photon}}}{\Delta\pT \times
  \Lint^{60\text{cm}} \times \varepsilon_{\text{reco}} \times
  \varepsilon_{\text{iso}} \times \varepsilon_{\text{id}} \times
  \varepsilon_{\text{vtx+MBD}}^{\text{old}}}
\end{equation}
where
\begin{equation}
  \varepsilon_{\text{vtx+MBD}}^{\text{old}} = \frac{N(\text{truth photons, }|z_{\text{truth}}|<60, \text{MBD+}|z_{\text{reco}}|<60)}{N(\text{truth photons, }|z_{\text{truth}}|<60)}
\end{equation}
The luminosity $\Lint^{60\text{cm}}$ is the value from
\texttt{60cmLumi\_fromJoey.list}, which is computed by
\texttt{get\_luminosity\_182630.C} with \texttt{zsel=1} ($|z_{\text{vtx}}|<60$~cm
applied at the scaler tally level).

\textbf{Alternative (no-vertex-cut convention):}
\begin{equation}
  \frac{d\sigma}{d\pT} = \frac{N_{\text{photon}}}{\Delta\pT \times
  \Lint^{\text{noVtx}} \times \varepsilon_{\text{reco}} \times
  \varepsilon_{\text{iso}} \times \varepsilon_{\text{id}} \times
  \varepsilon_{\text{vtx+MBD}}^{\text{new}}}
\end{equation}
where
\begin{equation}
  \varepsilon_{\text{vtx+MBD}}^{\text{new}} = \frac{N(\text{truth photons, }|z_{\text{truth}}|<60, \text{MBD+}|z_{\text{reco}}|<60)}{N(\text{truth photons, no vertex cut})}
\end{equation}
The luminosity $\Lint^{\text{noVtx}}$ is from \texttt{allzLumi\_fromJoey.list}
(\texttt{zsel=5}, no vertex cut at the scaler level).

The numerators of the two efficiency factors are \textbf{identical}.
Only the denominator differs: original uses truth photons with
$|z_{\text{truth}}|<60$, new uses ALL truth photons.

\textbf{Conceptual equivalence}: in the single-interaction limit, with
MC truth vertex distribution matching data, the two products should
satisfy:
\begin{equation}
  \Lint^{\text{noVtx}} \times \varepsilon_{\text{vtx+MBD}}^{\text{new}}
  \approx
  \Lint^{60\text{cm}} \times \varepsilon_{\text{vtx+MBD}}^{\text{old}}
  \label{eq:equiv}
\end{equation}
because $\Lint^{60\text{cm}} / \Lint^{\text{noVtx}} = $ data fraction with
$|z_{\text{vtx}}|<60$, and the same fraction should appear in the MC truth
vertex distribution after reweighting.

\section{Diagnosis: MC Vertex Reweighting}
\label{sec:diagnosis}

The discrepancy comes from how the MC vertex reweighting interacts with
the two efficiency denominators.

The production vertex reweighting in
\texttt{RecoEffCalculator\_TTreeReader.C} uses the \emph{reconstructed}
vertex (\texttt{*vertexz}) to compute a per-event weight that brings the
MC reco vertex distribution into agreement with data. The weight is
applied to all histogram fills, including \texttt{h\_truth\_pT} (line
1691, no vertex cut) and \texttt{h\_truth\_pT\_vertexcut} (line 1696,
with $|z_{\text{truth}}|<60$).

\textbf{The bug}: at line 1350, when the reweight histogram bin content
is non-finite or $\leq 0$ (which happens at large $|z_{\text{reco}}|$
where data has very few events), the code falls back to
\texttt{vertex\_weight = 1.0}. Events at large $|z_{\text{reco}}|$
(and correspondingly large $|z_{\text{truth}}|$ for single-interaction
MC) thus contribute with full weight to \texttt{h\_truth\_pT}, even though
they should be downweighted to match data.

The result: the integral of \texttt{h\_truth\_pT} is bloated by raw MC
events at $|z_{\text{truth}}| > 60$~cm, while
\texttt{h\_truth\_pT\_vertexcut} is unaffected (the vertex cut already
excludes these events). The ratio
\texttt{h\_truth\_pT\_vertexcut / h\_truth\_pT} drops to $\sim 0.36$
instead of the expected $\sim 0.98$ (for 1.5~mrad data, where the
true beam vertex distribution has $\sigma \sim 21$~cm).

\textbf{Why this doesn't affect the OLD path}: the OLD efficiency uses
\texttt{h\_truth\_pT\_vertexcut} in the denominator, which restricts to
$|z_{\text{truth}}|<60$. Within this range, the vertex reweighting
behaves correctly because the reweight histogram is well-defined
(both data and MC have ample events at $|z|<60$). The OLD path is
thus self-consistent and matches the
$\Lint^{60\text{cm}}$ luminosity correctly.

\section{Implementation}
\label{sec:implementation}

The alternative convention is wired into
\texttt{efficiencytool/CalculatePhotonYield.C} as a config-switchable
opt-in path. The default is the original convention.

\textbf{Code change}: lines 137--162 read a new config parameter
\texttt{use\_lumi\_novtx} (default 0). Lines 1063--1071 select the
denominator histogram based on this flag:

\begin{verbatim}
if (use_lumi_novtx && h_truth_pT_all)
{
    vertex_eff_val = h_truth_pT_vertexcut_mbd_cut->GetBinContent(ibin)
                   / h_truth_pT_all->GetBinContent(ibin) + mbd_eff_scale;
}
else
{
    vertex_eff_val = h_truth_pT_vertexcut_mbd_cut->GetBinContent(ibin)
                   / h_truth_pT_vertexcut->GetBinContent(ibin) + mbd_eff_scale;
}
\end{verbatim}

The new histogram \texttt{h\_truth\_pT\_all} is loaded from the existing
production output (already filled at \texttt{RecoEffCalculator\_TTreeReader.C}
line 1691, no MC re-run needed).

To use the new convention, set:
\begin{verbatim}
analysis:
  use_lumi_novtx: 1
  lumi: 17.1642   # for 1.5 mrad, from allzLumi_fromJoey.list
\end{verbatim}

The change is one line in the efficiency loop and one line in the config.

\section{Recommendation}
\label{sec:recommendation}

\subsection{Root cause: the static reweight fallback file is outdated}

A standalone Python validation using the current first-pass vtxscan
files shows that the \emph{on-the-fly} reweight path
(\texttt{RecoEffCalculator\_TTreeReader.C} lines 184-207, computed
from \texttt{h\_vertexz} in the per-config vtxscan ROOT files)
correctly reproduces the data vertex shape: after reweighting,
\texttt{(MC weighted within $|z|<60$) / (MC weighted total) = 0.987},
exactly matching the data target of 0.987.

The 0.355 ratio observed in
\texttt{results/MC\_efficiency\_nom.root} (June 2025) instead comes
from the \emph{file-fallback} path, which loads a static reweight
histogram \texttt{results/vertex\_reweight\_bdt\_none.root}. This file
was produced from an older, wider data vertex distribution and has
\texttt{min ratio = 0.49}, \texttt{max = 1.78} --- nearly flat, so MC
events at large $|z|$ are kept at $\sim 50\%$ weight rather than
being suppressed to $\sim 4\%$.

\textbf{Fix}: re-run the nominal pipeline with current vtxscan files
so the on-the-fly path is used (the file fallback is only triggered
when vtxscan files are absent). After re-running, the production
\texttt{h\_truth\_pT} integral should give the expected $\sim 0.987$
ratio against \texttt{h\_truth\_pT\_vertexcut}, and the no-vertex-cut
convention will give a cross-section consistent with the original
within $\leq 1\%$.

\subsection{Defensive code changes applied}

In addition to relying on the on-the-fly path, the following
robustness improvements are committed in
\texttt{RecoEffCalculator\_TTreeReader.C}:

\begin{enumerate}
  \item \textbf{Rebin before computing the ratio} (configurable,
    default factor 5 → 5~cm bins). This boosts statistics in the
    tails so the data/MC ratio is well-defined further out.

  \item \textbf{Smooth the resulting ratio} via \texttt{TH1::Smooth()}
    (configurable, default 1 pass) to remove residual statistical
    noise.

  \item \textbf{Linear interpolation between bin centers}
    (\texttt{TH1::Interpolate()}) for smooth lookup at event level,
    instead of nearest-bin lookup with \texttt{GetBinContent()}.

  \item \textbf{Zero fallback for out-of-range and undefined bins}
    (line 1350): changed from \texttt{vertex\_weight = 1.0} to
    \texttt{vertex\_weight = 0.0}. Events with $|z_{\text{reco}}|$
    outside the histogram range or where the ratio is non-finite are
    now \emph{dropped} (zero weight) instead of being kept at full
    weight.
\end{enumerate}

New config knobs:
\begin{verbatim}
analysis:
  vertex_reweight_rebin: 5       # rebin factor for h_vertexz before ratio
  vertex_reweight_smooth: 1      # number of TH1::Smooth() passes
\end{verbatim}

\subsection{Recommended action sequence}

\begin{enumerate}
  \item Re-run the nominal config (\texttt{oneforall.sh
    config\_bdt\_nom.yaml}) with the current code. The two-pass
    pipeline will use the on-the-fly reweight from fresh vtxscan
    files.
  \item Verify in the new \texttt{MC\_efficiency\_nom.root}:
    \begin{verbatim}
    h_truth_pT_vertexcut.Integral() / h_truth_pT.Integral() ~ 0.99 (1.5 mrad)
    \end{verbatim}
    If yes, the fix is confirmed.
  \item Compare the cross-section between the original convention
    ($\Lint^{60\text{cm}}$, \texttt{use\_lumi\_novtx: 0}) and the new
    convention ($\Lint^{\text{noVtx}}$, \texttt{use\_lumi\_novtx: 1}).
    Expected difference: $\leq 1\%$ at 1.5~mrad.
  \item If both give consistent results, adopt the no-vertex-cut
    convention as the default for cleaner physics interpretation.
\end{enumerate}

\textbf{Until then}, the production analysis should continue to use
the original convention (\texttt{use\_lumi\_novtx: 0},
$\Lint^{60\text{cm}}$).
