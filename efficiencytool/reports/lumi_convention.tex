\begin{abstract}
We switch the PPG12 cross-section normalization from the
60~cm-vertex-cut luminosity $\Lint^{60\text{cm}}$ (with a vertex
efficiency whose denominator is truth photons within
$|\zvtx^{\text{truth}}|<60$~cm) to the no-vertex-cut luminosity
$\Lint^{\text{noVtx}}$ (with an efficiency denominator that includes
ALL truth photons). The two conventions are mathematically equivalent
if the MC vertex reweighting correctly brings the MC vertex
distribution into agreement with data. In the old production, they
disagreed by a factor of $\sim 3$ because the MC vertex reweighting
used a deprecated static fallback file with a
\texttt{vertex\_weight = 1.0} fallback that kept raw MC events at
large $|\zvtx|$. We diagnosed the bug, removed the fallback entirely,
and hardened the on-the-fly reweight with rebinning, smoothing,
linear interpolation, and a \texttt{vertex\_weight = 0.0} fallback.
A full re-run of the nominal efficiency pipeline (photon5, photon10,
photon20 merged via \texttt{MergeSim.C}) with the fix applied
validates both periods separately: \textbf{1.5 mrad agrees to
$-0.34\%$ flat across all \pT\ bins} (integrated $L\times\veff$:
$15.33 \rightarrow 15.30$), while \textbf{0 mrad shows a $+14.16\%$
flat offset} (integrated $L\times\veff$: $29.29 \rightarrow 33.45$).
The 1.5 mrad agreement confirms the fix; the 0 mrad offset is not a
reweight bug but a real inconsistency between the lumi file's
scaler-level 60~cm fraction ($69.2\%$) and the slimtree's event-level
60~cm fraction ($79.1\%$, rerun-confirmed). This discrepancy must be
understood before adopting the new convention for 0~mrad.
\end{abstract}

\section{Motivation}
\label{sec:motivation}

The PPG12 cross-section factorizes as:
\begin{equation}
  \frac{d\sigma}{d\pT} = \frac{N_{\text{photon}}}{\Delta\pT \times \Lint \times \varepsilon_{\text{total}}},
  \qquad
  \varepsilon_{\text{total}} = \varepsilon_{\text{reco}} \times
  \varepsilon_{\text{iso}} \times \varepsilon_{\text{id}} \times \veff
\end{equation}

Two equivalent conventions are possible for $\Lint$ and $\veff$:

\textbf{Old convention} (used in production until now):
\begin{equation}
  \Lint = \Lint^{60\text{cm}}, \qquad
  \veff^{\text{old}} = \frac{N(\text{truth }\gamma,\ |z_t|<60,\ \text{MBD}+|z_r|<60)}{N(\text{truth }\gamma,\ |z_t|<60)}
\end{equation}
with $\Lint^{60\text{cm}}$ from \texttt{60cmLumi\_fromJoey.list}
(\texttt{get\_luminosity\_182630.C(..., zsel=1, ...)} with the MBD
scaler tally restricted to $|\zvtx|<60$~cm).

\textbf{New convention}:
\begin{equation}
  \Lint = \Lint^{\text{noVtx}}, \qquad
  \veff^{\text{new}} = \frac{N(\text{truth }\gamma,\ |z_t|<60,\ \text{MBD}+|z_r|<60)}{N(\text{truth }\gamma,\ \text{no vertex cut})}
\end{equation}
with $\Lint^{\text{noVtx}}$ from \texttt{allzLumi\_fromJoey.list}
(\texttt{zsel=5}, no vertex cut at the scaler level).

The two must agree (single-interaction limit, correct vertex
reweighting) to $\leq 1\%$ in every \pT\ bin. Any larger deviation
points at a discrepancy between how data is counted in the lumi file
and how it is sampled in the slimtree.

%% ============================================================
\section{The Bug in the Old Production}
\label{sec:bug}

The June 2025 production file \texttt{MC\_efficiency\_nom.root}
has $h_{\text{truth vertexcut}} / h_{\text{truth}} = 0.355$, far
from the expected data value of $\sim 0.99$. The cause: the old
\texttt{RecoEffCalculator\_TTreeReader.C} fell back to a static
file \texttt{results/vertex\_reweight\_bdt\_none.root} when the
on-the-fly path failed. That file was built from an older, wider
data vertex distribution and has min/max ratio 0.49 / 1.78 (nearly
flat). Combined with the event-level lookup fallback
\texttt{vertex\_weight = 1.0}, raw MC events at large $|\zvtx|$ were
kept at $\sim 50$--$100\%$ weight instead of being suppressed,
bloating the $h_{\text{truth pT}}$ denominator by a factor of
$\sim 2.8$.

%% ============================================================
\section{The Fix}
\label{sec:fix}

In commits \texttt{8ac4d1b} and \texttt{8ecae87}:

\begin{enumerate}
  \item \textbf{Static fallback removed} from both
    \texttt{RecoEffCalculator\_TTreeReader.C} and
    \texttt{ShowerShapeCheck.C}. If the on-the-fly vtxscan files
    are missing, the code prints \texttt{FATAL} and returns. The
    static file was renamed to \texttt{*.deprecated} on disk.
  \item \textbf{On-the-fly reweight hardened}:
  \begin{itemize}
    \item Rebin \texttt{h\_vertexz} (default factor
      $5 \rightarrow 5$~cm bins, configurable via
      \texttt{vertex\_reweight\_rebin}) to improve tail statistics.
    \item Smooth the data/sim ratio with \texttt{TH1::Smooth()}
      (default 1 pass, configurable via
      \texttt{vertex\_reweight\_smooth}).
    \item \texttt{TH1::Interpolate()} for smooth event-level lookup
      between bin centers.
    \item Fallback changed from \texttt{1.0} to \texttt{0.0}.
      Events outside the data vertex acceptance are dropped, not
      kept at full weight.
  \end{itemize}
  \item \textbf{New convention added as a config-switchable path}
    in \texttt{CalculatePhotonYield.C}
    (\texttt{use\_lumi\_novtx: 1}). Old convention remains the
    default. The alternative path only changes the denominator
    histogram (\texttt{h\_truth\_pT} instead of
    \texttt{h\_truth\_pT\_vertexcut}) and the luminosity value.
\end{enumerate}

\section{Analysis of the 0 mrad Discrepancy}
\label{sec:0mrad_discrepancy}

The $+14.2\%$ shift at 0 mrad is \emph{not} a bug in the reweight
fix --- both the old and new reweights reproduce the slimtree data
vertex distribution to $< 0.1\%$ (Figure~\ref{fig:vertex_dist} right
panels), and the production rerun gives
$h_{\text{truth pT vertexcut}}/h_{\text{truth pT}} = 0.7908$ in the
merged MC, confirming the fix is functioning at the histogram level.
The issue is a mismatch between two independent measurements of the
same quantity:

\begin{itemize}
  \item \textbf{Lumi file}: $\Lint^{60\text{cm}} / \Lint^{\text{noVtx}}
    = 32.70 / 47.23 = 0.6924$. This is the scaler-level ratio: the
    fraction of MBD-triggered events with the on-line
    $|z^{\text{MBD}}_{\text{vtx}}| < 60$~cm tally.
  \item \textbf{Slimtree vtxscan}: 0.7880. This is the fraction of
    events in the data slimtree that have
    $|z^{\text{reco}}_{\text{vtx}}| < 60$~cm.
  \item \textbf{Production MC (after fix)}: 0.7908. The reweighted
    MC truth-level ratio confirms the slimtree target is being
    reproduced in the full pipeline.
\end{itemize}

These numbers differ by 14.2\% for 0 mrad but agree to 0.2\% for
1.5 mrad. Possible explanations:

\begin{enumerate}
  \item \textbf{Slimtree event selection biases vertex distribution}:
    \texttt{CaloAna24.cc} saves an event only if it has clusters or
    particles (\texttt{saveevent = true}). At 0 mrad, the much wider
    vertex distribution means events at large $|\zvtx|$ have
    relatively fewer calorimeter clusters (forward activity is
    suppressed by the barrel-only acceptance), so they are more
    likely to be rejected by the slimtree filter. This preferentially
    removes large-$|\zvtx|$ events from the slimtree sample, giving
    the slimtree a more central distribution than the raw MBD tally.
  \item \textbf{Run list mismatch}: the 0 mrad lumi file and the
    0 mrad slimtree sample may not use exactly the same run list.
    A run-by-run comparison would be needed to rule this out.
  \item \textbf{Offline vs online vertex reconstruction}: the
    scaler tally uses the online MBD vertex. The slimtree uses
    \texttt{MbdVertex::get\_z()} from an offline reconstruction pass,
    which may have different calibration or resolution.
\end{enumerate}

The most likely explanation is (1) --- the slimtree filter
preferentially keeps central events at 0 mrad where the beam is
wider. At 1.5 mrad the beam is already narrow, so almost all events
are within the slimtree's effective acceptance and no bias is
introduced.

\subsection{Implication for the cross-section}

If explanation (1) is correct, then neither convention is
exactly right without a correction:

\begin{itemize}
  \item \textbf{OLD convention} uses $\Lint^{60\text{cm}}$ which
    counts events in the full MBD sample with $|\zvtx|<60$. But the
    numerator $N_{\text{photon}}$ comes from the slimtree, which
    excludes some of the large-$|\zvtx|$ events that are in the
    scaler tally. So OLD over-counts luminosity for the slimtree
    sample.
  \item \textbf{NEW convention} uses $\Lint^{\text{noVtx}}$ with
    $\veff^{\text{new}}$ computed from the slimtree vertex
    distribution. The slimtree is a biased (more central) subsample
    of the full data, so NEW gives a larger effective luminosity
    --- but only to the extent that it credits the slimtree bias
    as a legitimate physics acceptance.
\end{itemize}

The truth lies somewhere in between. The correct answer depends on
how the slimtree filter interacts with the photon selection. If the
slimtree filter removes events that genuinely have no
photons (e.g., events with no EMCal clusters because they are far
from the barrel), then those events contribute zero to
$N_{\text{photon}}$ anyway and the NEW convention is correct. If
the slimtree filter removes events that could have contained a
photon but didn't produce a cluster for reconstruction reasons, then
there is a real bias that neither convention captures.

\subsection{Recommended action}

\begin{enumerate}
  \item \textbf{Use the NEW convention for 1.5 mrad} (the nominal
    1.5 mrad result). The $-0.34\%$ shift is well within any
    systematic, and the per-$\pT$ ratio (Figure~\ref{fig:ratio}
    bottom-left) is flat within bin-to-bin scatter.
  \item \textbf{Do NOT use the NEW convention for 0 mrad} until
    the slimtree-vs-lumi-file discrepancy is resolved. The $+14.2\%$
    shift is too large to adopt without understanding.
  \item \textbf{Investigate the 0 mrad slimtree filter}: run the
    vtxscan on a no-cluster-filter slimtree (if one exists), or
    replay \texttt{CaloAna24.cc} with the \texttt{saveevent}
    condition relaxed, and compare the vertex distribution to the
    production slimtree.
  \item \textbf{Cross-check against per-run lumi}: sum
    $\Lint^{60\text{cm}}(\text{run})$ for runs that appear in the
    slimtree and compare to the total from the lumi file. This
    would rule out run-list mismatch (explanation~2).
\end{enumerate}

\section{Implementation and Usage}
\label{sec:implementation}

To enable the new convention for a specific analysis config:

\begin{verbatim}
analysis:
  use_lumi_novtx: 1
  lumi: 17.1642   # 1.5 mrad from allzLumi_fromJoey.list
  # use 47.2284 for 0 mrad, 64.3718 for combined
\end{verbatim}

The default is \texttt{use\_lumi\_novtx: 0} (old convention). Both
paths live in \texttt{CalculatePhotonYield.C} lines 1063--1071; the
original \texttt{vertex\_eff} calculation is kept as the default.

New config knobs for the vertex reweight (sensible defaults, existing
configs unchanged):
\begin{verbatim}
analysis:
  vertex_reweight_rebin: 5       # rebin factor (1cm -> 5cm)
  vertex_reweight_smooth: 1      # TH1::Smooth() passes
\end{verbatim}

The old \texttt{vertex\_reweight\_file} knob is no longer read.

\section{Caveats}
\label{sec:caveats}

\begin{enumerate}
  \item \textbf{0 mrad ambiguity} (Section~\ref{sec:0mrad_discrepancy}):
    do not adopt the new convention for 0 mrad without resolving
    the 14.2\% shift first.
  \item \textbf{Per-segment variation}: within a fill, the luminous
    region moves. This is typically $< 1\%$ effect at RHIC and is
    not addressed by the current period-level reweight.
  \item \textbf{Pileup}: the double-interaction MC blending in
    \texttt{run\_showershape\_double.sh} carries its own correction
    and is unaffected by this change.
  \item \textbf{Jet sample reuse}: the production rerun used the
    existing jet12/jet20/jet30/jet40 MC files (from 2025-04-11),
    which already used the post-fix reweight logic. Only the
    photon samples were re-run from scratch as part of this
    validation.
  \item \textbf{Full pipeline rerun}: in addition to the
    \texttt{RecoEffCalculator\_TTreeReader.C} photon jobs,
    \texttt{MergeSim.C}, the data Pass~2 jobs
    (\texttt{RecoEffCalculator\_TTreeReader.C} on data, both periods),
    and \texttt{CalculatePhotonYield.C} were also re-run to regenerate
    \texttt{Photon\_final\_bdt\_nom.root} and
    \texttt{Photon\_final\_bdt\_0rad.root} (both default, old
    convention). The updated final files are available for use in
    downstream plotting and systematics aggregation.
\end{enumerate}
